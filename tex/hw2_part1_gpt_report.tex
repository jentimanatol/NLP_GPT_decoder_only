\begin{abstract}
This report documents the implementation and testing of a small GPT-style decoder-only Transformer model for character-level next-token prediction. The project focuses on the first part of the assignment: building an autoregressive language model using causal self-attention, training it on a text dataset, saving model artifacts, and generating text from a prompt. A quick CPU training script was added to make the project reproducible and easy to test without requiring long training time or a GPU.
\end{abstract}

\section{Project Goal}

The goal of this project is to implement and test a decoder-only Transformer similar in structure to a small GPT model. The model receives a sequence of previous tokens and predicts the next token. Because the model is autoregressive, each token is allowed to attend only to itself and previous tokens, not to future tokens.

The main objectives are:

\begin{itemize}[leftmargin=*]
    \item implement causal self-attention;
    \item train the model using the next-token prediction objective;
    \item evaluate the model using validation loss and perplexity;
    \item save model artifacts for reproducibility;
    \item test text generation from a starting prompt.
\end{itemize}

\section{Model Overview}

The implemented model is a small decoder-only Transformer. It uses token embeddings, positional embeddings, masked self-attention blocks, feed-forward layers, normalization, and a final linear projection to predict the next character/token.

\subsection{Decoder-Only Architecture}

A decoder-only Transformer processes an input sequence:

\[
    x_1, x_2, \dots, x_T
\]

and predicts the next-token targets:

\[
    x_2, x_3, \dots, x_{T+1}.
\]

The model learns the probability distribution:

\[
    P(x_{t+1} \mid x_1, x_2, \dots, x_t).
\]

This is the standard autoregressive language modeling setup. The model does not use an encoder because the task is not sequence-to-sequence translation. Instead, it learns to continue text based only on previous context.

\section{Causal Self-Attention}

The most important part of the decoder-only model is causal self-attention. In normal self-attention, every token could attend to every other token. For language generation, that would be wrong because the model would be able to look at future answers during training.

To prevent information leakage, a causal mask is applied. The mask blocks attention from position \(t\) to any future position \(j > t\). The attention computation is:

\[
    \text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}} + M\right)V
\]

where \(M\) is the causal mask. Future positions receive a very negative value before the softmax, forcing their attention probability close to zero.

This means token \(t\) can only use:

\[
    x_1, x_2, \dots, x_t
\]

and cannot use:

\[
    x_{t+1}, x_{t+2}, \dots, x_T.
\]

\section{Training Objective}

The model is trained using the next-token prediction objective. For each input sequence, the target is the same sequence shifted one position to the left.

For example, if the input sequence is:

\begin{lstlisting}
ROMEO:
\end{lstlisting}

then the model learns to predict each next character after seeing the previous characters.

The loss function is cross-entropy loss:

\[
    \mathcal{L} = - \frac{1}{N}\sum_{i=1}^{N} \log P(y_i \mid x_{\leq i})
\]

where \(y_i\) is the correct next token and \(P(y_i \mid x_{\leq i})\) is the probability assigned by the model.

\section{Quick CPU Training Script}

The original full epoch training script was too slow and did not show progress frequently enough on CPU. To solve this, a quick training script was added. Instead of training by full epochs, it trains for a fixed number of random mini-batch steps. This makes the project easier to test on a normal laptop.

The quick training script provides:

\begin{itemize}[leftmargin=*]
    \item fixed-step training instead of full-epoch training;
    \item visible progress heartbeat messages;
    \item validation checks during training;
    \item automatic best-model checkpoint saving;
    \item final result files for the report;
    \item generation test after training.
\end{itemize}

The training batch file runs the quick training script and then tests generation. This makes the project easier for the instructor or another user to reproduce.

\section{How to Run the Project}

From the project directory, activate the Python environment if needed and run:

\begin{lstlisting}[language=bash]
training.bat
\end{lstlisting}

The batch file starts quick CPU training and then runs a generation test.

A direct Python command can also be used:

\begin{lstlisting}[language=bash]
python train_quick_cpu.py --steps 1200
\end{lstlisting}

After training, generated files are stored in the \texttt{outputs} directory.

\section{Generation Test}

After training, the model was tested using a prompt beginning with \texttt{ROMEO:}. The generated output was not perfect English, but it showed that the complete pipeline works: the trained checkpoint can be loaded, the prompt can be encoded, the model can autoregressively sample next tokens, and the generated tokens can be decoded back into text.

Example generated text from the quick run:

\begin{lstlisting}
ROMEO:
TRI:
MGO:
D ce cendoulrthiat be, bne hrino ou ir dofourthite ive berel,

Whye t to s wimas lllf ck reare re itonouleas lir ha f inat toreara sthar y
\end{lstlisting}

The output is expected to be noisy because the model is intentionally small and trained briefly on CPU. The purpose of the quick run is reproducibility and verification, not production-quality text generation.

\section{Discussion}

The training results show that the decoder-only Transformer implementation is functioning correctly. The validation perplexity decreased from about 75.50 to 12.01, which is a strong sign that the model learned useful character-level patterns from the dataset.

The quick CPU training version is useful for a homework submission because it avoids long full-epoch training and gives immediate evidence that the code works. It also prints regular progress messages so the user can see that training is active.

For better generated text, the model could be trained longer, increased in size, or trained on a larger dataset. However, for this assignment, the current result is enough to demonstrate the core concepts:

\begin{itemize}[leftmargin=*]
    \item causal masking prevents future-token leakage;
    \item next-token prediction trains the language model;
    \item the validation loss and perplexity improve during training;
    \item the model can save checkpoints and generate text.
\end{itemize}

\section{Limitations}

The current implementation is intentionally small and CPU-friendly. Because of that, the generated text is still noisy and not fully coherent. The main limitations are:

\begin{itemize}[leftmargin=*]
    \item small number of trainable parameters;
    \item character-level tokenization instead of subword tokenization;
    \item short training time;
    \item CPU-only training;
    \item limited dataset size.
\end{itemize}

These limitations are acceptable for a classroom demonstration because the goal is to show a working decoder-only Transformer training pipeline, not to build a large production language model.

\section{Conclusion}

This project successfully implements and tests a GPT-style decoder-only Transformer for next-token prediction. The model uses causal self-attention to prevent access to future tokens and learns through cross-entropy loss. A quick CPU training script was added to make the project reproducible and easy to test. The final quick run achieved a best validation NLL of 2.4855 and a best validation perplexity of 12.01, confirming that the training process works correctly.
